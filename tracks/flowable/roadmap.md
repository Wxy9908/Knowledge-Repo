# Flowable 流程对接 — 学习路线（前端视角）

> **角色定位**：流程**搭建**由后端负责；你学 Flowable 是为了**对接时不懵**——能看懂流程、理解接口、开发出审批相关页面、联调时能沟通排错。  
> **学习边界**：应用层掌握，**不读源码**；**不以自己搭引擎、画流程为主**。  
> **建议周期**：3–4 周（每周 4–6 小时）  
> **前置**：Vue 3 基础、`ai-coding` 中的接口请求与页面开发经验

## 学习目标

**最终目标**：后端给你一份请假/报销类流程的接口文档和 BPMN 图时，你能：

1. **看懂**流程大致怎么走、当前可能卡在哪
2. **开发**发起页、待办列表、审批操作、流程进度/历史展示
3. **联调**时知道传什么字段、报错时往前端还是后端方向查
4. **沟通**时能使用正确术语，不把「任务」「实例」「业务单」混为一谈

**核心标准**：**会读、会接、会展示、会联调** —— 不是「会搭流程」。

---

## 前后端职责边界

| 事项 | 后端 | 你（前端） |
|------|------|------------|
| BPMN 画图、部署流程 | ✅ 主责 | 能**看懂**即可，不要求独立建模 |
| 启动 Flowable、引擎配置 | ✅ 主责 | 知道引擎存在，不自己搭 |
| 封装业务 API（发起、审批、查询） | ✅ 主责 | **消费**这些 API |
| 流程变量、网关条件、办理人规则 | ✅ 主责 | 知道变量要从表单传过去、条件影响分支 |
| 发起表单、待办列表、审批页 UI | 配合 | ✅ **主责** |
| 流程状态、审批时间线展示 | 配合 | ✅ **主责** |
| 监听器、回写业务表 | ✅ 主责 | 知道「流程结束后业务状态会变」，前端刷新或轮询 |

**你不必学的内容**（除非工作明确要求）：Spring Boot 集成、流程部署、写 Java 监听器、Flowable 源码。

## 能力地图（前端版）

| # | 能力 | 掌握标志 |
|---|------|----------|
| 1 | **BPMN 阅读** | 看到流程图能说出：有哪些审批节点、同意/驳回往哪走 |
| 2 | **核心概念** | 说清：业务单 vs 流程实例 vs 待办任务 vs 流程变量 |
| 3 | **接口契约** | 知道典型接口：发起、我的待办、完成任务、流程详情、审批历史 |
| 4 | **页面开发** | 能做出：发起页 + 待办列表 + 审批抽屉/页 + 状态/时间线 |
| 5 | **数据映射** | 表单字段 ↔ 流程变量；列表项展示哪些后端字段 |
| 6 | **联调排错** | 任务 404、无待办、状态不同步时，能判断查接口文档还是找后端 |

## 三阶段学习路径

```
Phase 0 概念期（约 1 周）   →  术语 + 看懂 BPMN，零代码
```

### Phase 0：概念期 —— 能参与需求评审

**阶段目标**：后端讲流程时你能跟上，看 PRD 里「审批流」不发懵。

| 学习内容 | 产出 |
|----------|------|
| 审批类业务为什么用流程引擎（对比 if/else） | 笔记 `notes/bpmn-basics.md` |
| BPMN 图**阅读**：用户任务、网关、结束（不用精通建模） | 能解读一张请假流程图 |
| 四个词：流程定义、流程实例、用户任务、流程变量 | 笔记 `notes/flowable-architecture.md` |
| `businessKey`：业务单号和流程实例怎么关联 | 一段话 + 示意图 |

**通关标准**

- [ ] 后端说「实例卡在经理审批节点」你能理解
- [ ] 能区分「业务单已保存」和「流程已发起」

---

### Phase 1：接口期 —— 搞懂你要调什么

**阶段目标**：不依赖 Flowable 自带 UI，清楚前端每条链路调哪个**业务 API**（通常由后端封装，不直接裸调引擎）。

#### 典型接口清单（与后端对齐用）

| 场景 | 常见接口语义 | 前端关心 |
|------|--------------|----------|
| 员工发起请假 | `POST /leave` 或 `POST /workflow/start` | 提交表单字段；返回业务 id / 实例 id |
| 我的待办 | `GET /tasks/todo` | 列表字段：标题、发起人、时间、taskId |
| 审批 | `POST /tasks/{id}/complete` | 同意/驳回、意见、必要时带变量 |
| 我发起的 | `GET /process/instances/mine` | 当前状态、进行到哪一步 |
| 审批历史 | `GET /process/instances/{id}/history` | 时间线：谁、何时、结果、意见 |
| 详情 | `GET /leave/{id}` | 业务数据 + 流程状态（可能合并返回） |

#### 练习 1：接口文档阅读

- 拿一份真实或示例 API 文档（可让后端提供，或用笔记里的 mock 契约）
- 列出：每个页面需要调哪些接口、字段含义

**验收**：画出「发起页 / 待办页 / 详情页」与接口的对应表

#### 练习 2：Postman 或 Mock 走通主流程

- 用 Postman 或 `msw`/json mock 模拟：发起 → 查待办 → 审批 → 查历史
- **不必自己部署 Flowable**；重点是熟悉请求/响应结构

**验收**：能口述一条请假从提交到审批结束的数据流

#### 练习 3：字段与状态映射

- 业务状态（草稿 / 审批中 / 已通过 / 已驳回）与流程状态的关系
- 驳回后：重新编辑再提交，还是原实例继续（与后端确认规则）

**验收**：写一张「前端展示状态 ↔ 后端字段」对照表

**Phase 1 通关标准**

- [ ] 完成练习 1–3
- [ ] 拿到接口文档后 30 分钟内能列出前端开发任务清单

### Phase 2：实战期 —— Vue 页面开发（重心）

**阶段目标**：开发出可演示的审批相关前端模块；这里最贴合你日常工作。

#### 练习 4：发起页（请假申请）

- 表单：天数、原因、附件（可选）
- 提交后跳转「我发起的」或详情页
- 处理 loading、校验、错误提示

**验收**：`pnpm dev` 可演示发起；mock 或真实接口均可

#### 练习 5：待办中心

- 待办列表：标题、发起人、到达时间、摘要
- 点击进入审批：同意 / 驳回 + 审批意见
- 空状态、无权限提示

**验收**：在浏览器完成一次完整审批操作

#### 练习 6：详情 + 审批历史时间线

- 业务信息展示区
- 流程进度：当前节点、已完成节点
- 时间线：每步审批人、时间、结果、意见

**验收**：审批前后状态与时间线正确更新

#### 练习 7（可选）：我发起的 / 已办

- Tab：待办 / 已办 / 我发起的
- 筛选、分页（与后端能力对齐）

**Phase 2 通关标准**

- [ ] 完成练习 4–6（7 可选）
- [ ] Demo 可放在 `playground/vue-playground/src/demos/flowable/`
- [ ] 写过联调复盘：遇到过哪些接口/状态问题

## 与后端协作：说什么话

联调时你可以主动确认：

| 问题 | 为什么前端需要知道 |
|------|-------------------|
| 发起接口一次提交哪些字段？哪些进流程变量？ | 表单设计和校验 |
| 待办列表每条的任务 id 字段名？ | 审批提交 |
| 同意/驳回是同一个接口还是两个？要传 `approved: true/false` 吗？ | 按钮逻辑 |
| 驳回后业务单状态是什么？能否编辑再提？ | 详情页按钮显隐 |
| 流程详情和历史是分开接口还是合并？ | 页面请求设计 |
| `businessKey` 用业务单 id 吗？ | 跳转、详情查询 |
| 当前用户身份怎么传？任务为何查不到？ | 待办为空排查 |

## 与现有轨道的衔接

| 你的现状 | 建议 |
|----------|------|
| `ai-coding` 接近通关 | Phase 2 直接用 Prompt 工作流做页面，和日常开发一致 |
| `fullstack-transition` | 只需 **REST 请求 + 类型定义** 即可，不必等为 Flowable 去学 Java |
| 公司暂无 Flowable 环境 | Phase 0–1 用文档 + Mock；Phase 2 用 mock 数据照样练页面 |

**推荐开轨时机**：**现在就可以 Phase 0**；有 Vue 基础后随时进入 Phase 2，不依赖后端轨道进度。

## 常见踩坑（前端向）

| 现象 | 常见原因 | 你怎么查 |
|------|----------|----------|
| 待办列表为空 | 当前用户不是 assignee；或后端按候选组过滤 | 让后端查任务归属；确认登录用户 id |
| 审批提交 404 | taskId 过期或已被他人处理 | 刷新列表；提示用户任务已处理 |
| 详情状态不对 | 只看业务表未同步流程状态 | 确认接口是否返回流程字段；审批后是否需刷新 |
| 发起成功但无待办 | 办理人配置错误 | **后端问题**，你提供发起参数即可 |
| 驳回后按钮乱了 | 未对齐「驳回后可否编辑」规则 | 产品 + 后端确认，前端按状态机显隐 |

## 可选附录：本地想自测又没有后端时

仅在这种情况下做，**不是前端必修**：

1. 用 Docker 跑 Flowable REST，或请后端搭一个最小 demo 服务
2. 走通一次「部署流程 → 启动 → 完成」加深理解
3. 目的：联调时更理解后端在说什么，**不是为了替代后端职责**
