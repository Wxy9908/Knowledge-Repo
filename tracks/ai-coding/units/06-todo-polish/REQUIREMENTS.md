## 1. 产品背景

待办应用 v1.0 已上线。产品反馈：

1. 误点「完成」无法改回
2. 标题打错字只能删了重建
3. 筛选结果为空时页面一片空白，用户不知道是没数据还是 bug

v1.1 要在 **不破坏登录、守卫、持久化** 的前提下迭代。

## 2. 功能需求

### 2.1 必做（P0 · 全部交付才算通过）

| #   | 功能             | 描述                                                                     |
| --- | ---------------- | ------------------------------------------------------------------------ |
| F1  | **切换完成状态** | 每条 todo 可在「已完成 ↔ 未完成」之间**切换**（不是只能完成一次）        |
| F2  | **编辑标题**     | 每条 todo 可修改 `title`；**非空**才保存，与新增规则一致                 |
| F3  | **空状态**       | 当前筛选下 **没有任何条目** 时，列表区域展示可读提示（文案自定，需可见） |
| F4  | **持久化兼容**   | F1、F2 改动后仍写入 `app-todos`；刷新后状态正确                          |

### 2.2 可选（P1 · 时间够再做）

| #   | 功能                  | 描述                                                               |
| --- | --------------------- | ------------------------------------------------------------------ |
| O1  | **清除已完成**        | 一键删除所有 `done: true` 的项（需二次确认或明确按钮文案，防误触） |
| O2  | **Header 显示用户名** | 已登录时在 `AppHeader` 展示 `currentUser.username`                 |

### 2.3 明确不做

- 不改登录 Mock 规则、不加真实接口
- 不持久化 `filter`、不持久化 `user`
- 不引入新 npm 依赖
- 不做拖拽排序、截止日期、标签

### 实现思路

F1：将`完成 completeTodo`方法改成切换逻辑，字段展示为 状态：已完成 -> 回退，状态：未完成 -> 完成，同时按钮的颜色不同，以作区分
F2：在标题右侧加一个`笔`的小图标，点击可将标题变成可输入状态，再次点击保存修改，校验逻辑与新增保存一致（建议使用一套校验逻辑，避免冗余代码），不支持Esc / Enter按钮，只支持鼠标操作。
F3: 类似el-empty对`todo-list`进行空状态的包裹，没有时候展示文案`暂无代办`并展示空图标。
F4: 将上述新操作写进`src/stores/todos.js`里，并支持持久化

## 4. 故意留白（考察你的假设 · 需写进 Prompt）

以下 **PRD 未规定**，你必须在 Prompt 的「不明确（假设）」里写清选择：

4. **文件范围**：是否只改 `todos.js` + `TodoList.vue`？

## 5. 验收场景（面试官会点）

```
□ 登录后进 /todos
□ 将一条「未完成」标为完成 → 再标回未完成（F1）
□ 修改某条标题 → 刷新 → 新标题仍在（F2 + F4）
□ 切到「已完成」且当前无已完成项 → 看到空状态提示（F3）
□ 在「未完成」筛选下新增/完成导致列表变空 → 空状态更新
□ 原有：增删、筛选、守卫、退出登录、持久化仍正常
□ （若做 O1）清除已完成后刷新数据一致
```
