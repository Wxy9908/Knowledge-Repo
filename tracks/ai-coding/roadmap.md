## 目标与标准

**最终目标**：在限时场景下（如 45–60 分钟面试），能够独立使用 AI 辅助，从需求文档交付**可运行、可演示、可解释**的前端功能。

**核心标准**：又快又好 —— 快是流程与迭代效率，好是功能正确、结构可控、稳定可验收。

---

## 一、你要掌握什么（能力地图）

完全掌握 AI Coding，不是「会跟 AI 聊天」，而是具备以下 **6 项能力**：

| # | 能力 | 说明 | 掌握标志 |
|---|------|------|----------|
| 1 | **需求拆解** | 从文档中提取必须/可选/不明确，列出页面、组件、路由、状态、接口 | 3 分钟内输出清晰清单，无明显遗漏 |
| 2 | **任务拆分** | 按依赖拆成 5–15 分钟可验收的小步 | 每步范围单一，不跨层（路由/组件/状态混在一起） |
| 3 | **Prompt 工程** | 用结构化 Prompt 约束 AI：背景、文件、边界、验收 | 复制模板后 2 分钟内改完并发出 |
| 4 | **边界把控** | 控制范围、技术、数据、交互、质量、时间六类边界 | AI 生成后 diff 可控，无无关改动 |
| 5 | **验收修复** | 跑通、测主流程、贴报错精准修复 | [功能 5 项](./notes/ai-coding-prompts.md#功能-5-项每步必查) + [体验 5 项](./notes/ai-coding-prompts.md#体验-5-项uiux-pass-或上线前) 形成习惯 |
| 6 | **表达复盘** | 向面试官说明设计决策，复盘优化 Prompt | 30 秒说清「做了什么、为什么、下一步」 |

## 二、标准工作流（每次练习/面试都走）

```
1. 读需求 → 2. 拆任务 → 3. 写 Prompt → 4. AI 生成 → 5. 验收修复 → 6. 复盘
```

| 步骤 | 你要做什么 | AI 能否辅助 | 速查 |
|------|------------|-------------|------|
| 1. 读需求 | 标注必须/可选/不明确；列清单 | ✅ 可让 AI 初稿，**你审核** | [Prompt 模板 §1](./notes/ai-coding-prompts.md#1-读需求分析step-1) |
| 2. 拆任务 | 定顺序、粒度、每步验收标准 | ✅ 可让 AI 建议，**你拍板** | [Prompt 模板 §2](./notes/ai-coding-prompts.md#2-拆任务step-2) |
| 3. 写 Prompt | 填模板：背景、需求、约束与范围（简单题可合并）、验收标准（2～3 条） | ✅ 可让 AI 起草，**你定稿** | [约束 FAQ Q4/Q5](./notes/ai-coding-prompts.md#约束-faq) |
| 4. AI 生成 | 扫 diff → 运行 → 不对则缩小范围重试 | — 主战场 | [Prompt 模板 §3](./notes/ai-coding-prompts.md#3-代码生成step-4--主模板) |
| 5. 验收修复 | [功能 5 项](./notes/ai-coding-prompts.md#功能-5-项每步必查) + 必要时 [UI Pass](./notes/ai-coding-prompts.md#7-uiux-pass功能验收后--体验层) | ✅ AI 分析，**你确认** | [体验 5 项](./notes/ai-coding-prompts.md#体验-5-项uiux-pass-或上线前) · [修 bug §4](./notes/ai-coding-prompts.md#4-修-bugstep-5) |
| 6. 复盘 | 记录有效/无效 Prompt，更新约束 | ✅ AI 总结，**你内化** | [复盘模板 §6](./notes/ai-coding-prompts.md#6-复盘step-6) |

## 三、代码生成时的六类边界（又快又好的关键）

每次 Step 4 生成前，Prompt 里至少明确前 3 类：

### 1. 范围边界（最重要）

- **本次只做** / **明确不做**
- **文件白名单**（允许改哪些）/ **黑名单**（禁止改 App.vue、package.json 等）

> 白名单/黑名单写法 → 见 [`AI coding.md` 约束 FAQ](./notes/ai-coding-prompts.md#约束-faq)

### 2. 技术边界

- Vue 3 + Composition API + `<script setup>`
- 页面 `src/views/`，组件 `src/components/`，状态 `src/stores/`
- 不引入新依赖（除非需求允许）
- 参考现有文件风格（如 `HomeView.vue`）

### 3. 数据边界

- 数据结构字段名（如 `{ id, title, done }`）
- 状态放哪（组件 ref / Pinia / localStorage）
- 空数据、非法输入怎么处理

### 4. 交互边界

- 本步支持哪些操作，不支持哪些
- 是否需要 loading、Toast、确认框（默认不做）

### 5. 质量边界（验收底线）

**功能 5 项**（每步必查）与 **体验 5 项**（UI Pass / 上线前）见 [`AI coding.md` 验收必查清单](./notes/ai-coding-prompts.md#验收必查清单)。

功能底线速查：

```
□ pnpm dev 能启动
□ 目标路由能访问，不白屏
□ 核心功能可用
□ 控制台无报错
□ 样式基本可读
```

### 6. 时间边界

- 单步 5–15 分钟；超时 → 缩小 scope，不堆功能
- **本步不过验收，不开下一步**

## 3.5 UI/UX 与 AI 协作（前端素养）

> **完整说明、UI Pass 模板、话术对照** → [`AI coding.md` § UI/UX 与 AI 协作](./notes/ai-coding-prompts.md#uiux-与-ai-协作)

前端不能只做功能不管展示。学习路线中的定位：

```text
层 1 · 工程层   功能 Prompt（第一刀）  →  Phase 1–3 各练习的核心
层 2 · 体验层   UI Pass / 对话迭代     →  练习 2 已实践；规范沉淀到设计资产
```

### 本项目设计资产（练习 2 起沉淀）

| 资产 | 位置 | Prompt 中如何引用 |
|------|------|-------------------|
| Design Tokens | `src/assets/base.css` | 「使用 `--color-*` / `--btn-*` / `--input-*`，禁止硬编码」 |
| 组件契约 | `src/assets/components.css` | 「按钮 `.btn .btn--primary`，输入 `.input`，状态 `.status-tag`」 |

### 练习中何时出现 UI/UX

| 练习 | 功能 Prompt | UI/UX 迭代 |
|------|-------------|------------|
| 练习 1 | 路由 + 列表 + 布局 | App 重构、nav 样式统一 |
| 练习 2 | 添加 / 删除 | 按钮色系、状态标签、系统色、全局 class、输入框 |
| 练习 3+ | Pinia、筛选等 | 功能 Prompt 引用 design system；必要时 UI Pass |

- [x] 功能验收后会做 **体验 5 项** 或 UI Pass（练 2 迭代 + 练 3 约束；Phase 2 制度化 UI Pass）

## 四、三阶段学习规划

```
Phase 1 跟练期（练习 1–3）  →  我带你逐步走，体会每步
Phase 2 半独立期（练习 4–5） →  你主导 Prompt，我关键点纠正
Phase 3 模拟面试（练习 6）   →  完整需求 + 限时，你独立完成
```

### Phase 1：跟练期 —— 建立流程肌肉记忆

**阶段目标**：熟悉 6 步工作流，会用 Prompt 模板，建立边界意识。

**形式**：每个练习分多次对话完成；每步我先示范 → 你确认/跟做 → 再进入下一步；**整个练习结束后统一复盘**。

#### 需要掌握的点

| 掌握点 | 具体标准 |
|--------|----------|
| 读需求 | 能独立标出必须/可选/不明确，列出 4 类清单（页面/组件/路由/状态） |
| 拆任务 | 第一步固定为「路由 + 骨架」，不一次做 CRUD |
| 写 Prompt | 会用模板；简单题合并「约束与范围」；会写 2～3 条验收标准 |
| 扫 diff | 生成后 30 秒内判断：文件是否越界、是否多做功能 |
| 验收 | 功能 5 项必查；UI 问题可对话修或走 UI Pass |

#### 练习清单

| 练习 | 需求概要 | 新增能力点 |
|------|----------|------------|
| **练习 1** | 静态 Todo 列表页（`/todos`，3 条静态数据） | 路由、View/Component 分层、nav、合并 Prompt |
| **练习 2** | 在练习 1 上增加添加/删除 Todo | 表单、事件、ref 状态、design system、UI Pass |
| **练习 3** | Pinia 管理 Todo + 筛选（全部/已完成/未完成） | Store 设计、computed 筛选 |

#### 每步跟练节奏（练习 1 示例）

| 步骤 | 内容 | 预计时间 |
|------|------|----------|
| Step 1 | 读需求，确认必须做/不做/文件清单 | 5 min |
| Step 2 | 拆成 3–4 个小任务 | 3 min |
| Step 3 | 写第 1 个 Prompt（路由+骨架） | 3 min |
| Step 4 | AI 生成，你扫 diff | 5 min |
| Step 5 | 验收，有问题则修复 Prompt 再生成 | 5–10 min |
| … | 重复 3–5，直到功能完整 | — |
| Step 6 | 练习结束复盘 | 5 min |

### Phase 2：半独立期 —— 自己驱动，关键处纠偏

**阶段目标**：你主导读需求、拆任务、写 Prompt；我只在范围失控、技术选型错误、验收遗漏时介入。

**形式**：我先给需求摘要 → **你先提交** Step 1 分析 + Step 2 任务清单 + Step 3 Prompt → 我点评后再生成代码。

| Prompt 定稿 | 含六类边界中的范围+技术+数据 |
| 调试反馈 | 报错信息 + @ 文件，一次修复到位 |

### Phase 3：模拟面试 —— 全真限时独立

**阶段目标**：模拟真实面试：完整需求文档、45–60 分钟、独立 AI Coding、可演示交付。

**形式**：

1. 我发**完整需求文档**（含模糊点，模拟真实文档）
2. 你限时独立完成（我仅在你说「提交」或时间到后复盘）
3. 结构化 debrief + 改进清单

**需求分析谁来做**：用 **需求拆解 Agent**（[`requirements-analyst.mdc`](../../_templates/agent-workflow/.cursor/rules/requirements-analyst.mdc) + [`AI coding.md` §0](./notes/ai-coding-prompts.md#0-需求拆解模板)）代替教练 Step 1；**模拟中途仍不问教练**。

#### 需要掌握的点

| 掌握点 | 具体标准 |
|--------|----------|
| 时间管理 | 前 5 min 分析+拆分，留 10 min 验收和收尾 |
| 优先级 | 必做先交付，可选后做；来不及则主动说明 |
| 完整交付 | 可 `pnpm dev` 演示主流程 |
| 专业表达 | 演示时说明：范围、假设、已知限制 |
| 抗压 | 中途报错不慌，缩小 scope 保主流程 |

## 五、Prompt 使用策略（文档 vs 现场）

| 层级 | 放哪里 | 用途 |
|------|--------|------|
| **骨架模板** | [`AI coding.md`](./notes/ai-coding-prompts.md#prompt-模板) | **7 类**场景模板，长期保留 |
| **每次任务** | 对话里复制改 3 处 | 需求、本次只做、文件路径、验收标准 |
| **进化约束** | [`AI coding.md` § 个人约束清单](./notes/ai-coding-prompts.md#个人约束清单) | 复盘后沉淀，如「不要改 App.vue」 |

**原则**：文档存结构，现场填内容；不要每次从零写，也不要不改动机械复制。

## 六、常见踩坑与对策

| 踩坑 | 对策 |
|------|------|
| 一次让 AI 做整个项目 | 拆 3–5 步，每步验收 |
| Prompt 太泛 | 加文件白名单 + 本次只做 |
| 不运行就继续 | [功能 5 项](./notes/ai-coding-prompts.md#功能-5-项每步必查) |
| AI 改无关文件 | 黑名单 + 扫 diff |
| 报错只说「不对」 | 贴完整报错 + @ 文件 |
| 追求样式忽略功能 | 先主流程；功能 OK 后 [UI Pass](./notes/ai-coding-prompts.md#7-uiux-pass功能验收后--体验层) 或对话迭代 |
| 样式全靠经验、不可复现 | 沉淀 tokens + 组件契约；见 [UI/UX 协作](./notes/ai-coding-prompts.md#uiux-与-ai-协作) |
| 约束 vs 本次只做混淆 | 见 [FAQ Q4](./notes/ai-coding-prompts.md#q4-约束和本次只做有什么区别会不会重复) |

**需求文档 v1.0 — Todo 列表页（静态展示）**

**1. 概述**  
新增「我的待办」页面，展示待办列表（本版本只读，不支持增删改）。

**2. 路由**  
- 路径：`/todos`  
- 导航：在顶部 nav 增加「Todos」链接  

**3. 页面**  
- 标题：「我的待办」  
- 列表：至少展示 3 条静态数据  
- 每条：标题 + 是否完成（完成显示「已完成」，未完成显示「未完成」）  

**4. 技术要求**  
- Vue 3 Composition API + `<script setup>`  
- 列表逻辑放在 `src/components/TodoList.vue`  
- 页面放在 `src/views/TodosView.vue`  
- 不引入新依赖  

**5. 不在本版本范围**  
- 添加 / 删除 / 编辑  
- Pinia  
- 接口请求  
- 样式精致化（基本可读即可）
