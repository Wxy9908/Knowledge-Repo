## 核心理念

**又快又好** —— 快是流程与迭代效率，好是功能正确、结构可控、稳定可验收。

---

## 标准工作流（速览）

```
1. 读需求 → 2. 拆任务 → 3. 写 Prompt → 4. AI 生成 → 5. 验收修复 → 6. 复盘
```

| 步骤         | 要点                                                  |
| ------------ | ----------------------------------------------------- |
| 1. 读需求    | 标必须/可选/不明确；列页面、组件、路由、状态、接口    |
| 2. 拆任务    | 按依赖拆小步；第一步通常是「路由 + 骨架」             |
| 3. 写 Prompt | 填模板：背景、需求、约束与范围（简单题可合并）、验收标准（2～3 条，可选） |
| 4. AI 生成   | 一次一小块；生成后先扫 diff，再运行                   |
| 5. 验收修复  | 跑 [必查清单](#验收必查清单)；报错贴完整信息 + @ 文件 |
| 6. 复盘      | 哪条 Prompt 有效？AI 哪里跑偏？约束如何加强？         |

> 工作流详解、六类边界、AI 辅助说明 → 见 [`AI-coding-学习路线.md` § 二、三](./AI-coding-学习路线.md#二标准工作流每次练习面试都走)

## 验收必查清单

### 功能 5 项（每步必查）

```
□ pnpm dev 能启动
□ 目标路由能访问，不白屏
□ 核心功能可用
□ 控制台无报错
□ 样式基本可读
```

### 体验 5 项（UI/UX Pass 或上线前）

```
□ 信息层级清晰：标题 > 操作区 > 内容列表
□ 操作可发现：主按钮 / 危险按钮位置一致
□ 状态可识别：完成 / 未完成等一眼能区分
□ 表单可用：focus 态、placeholder、与按钮等高对齐
□ 设计系统：无硬编码色值，复用全局 class 与 CSS 变量
```

反馈 AI 修复时：**贴完整报错信息**，并 `@` 相关文件。

## Prompt 模板

复制对应模板，改 3 处即可发出：**需求片段**、**本次只做**、**文件路径 / 验收标准**。

### 0. 需求拆解模板

> **角色**：需求拆解 Agent（`requirements-analyst.mdc`）  
> **用途**：原始 PRD / 面试题 → 结构化分析（类似 practice `README.md`）  
> **Phase 3**：模拟面试前 5～8 分钟专用；**不写 Prompt、不写代码**

```markdown
@tracks/ai-coding/STATUS.md
@_templates/agent-workflow/.cursor/rules/requirements-analyst.mdc

角色：需求拆解

## 原始需求

【粘贴需求全文】

## 请你输出（对齐 practice README 结构）

1. **需求摘要**（一句话）
2. **必须做** / **不做**
3. **不明确（我的假设）** — 列出 3～5 条合理假设供我确认
4. **涉及文件** — 页面 / 组件 / 路由 / store（新建 vs 修改）
5. **数据边界** — 字段结构、持久化、鉴权等
6. **建议 Prompt 拆步** — 1～3 步，每步：做什么 + 验收（可观察）
7. **验收清单** — 6～10 条 checkbox
8. **与现有代码衔接** — 基于 STATUS / 当前仓库，哪些可复用

## 约束

- 只输出分析，不写代码，不写 prompt.md
- 不代替我做最终优先级决策；冲突处列出选项
```

**用户后续**：把第 2～7 节改写成你的 `prompt.md`（或先存 `practice N/analysis.md` 再写 Prompt）。

### 1. 读需求分析（Step 1 · 轻量版）

> 极短需求可用本模板；**长 PRD / 模拟面试优先用 §0 需求拆解**。

```markdown
## 背景

Vue 3 + Vite 项目，【简述当前任务】

## 需求文档

【粘贴需求原文或链接】

## 请你帮我

1. 标注：必须做 / 可选做 / 不明确
2. 列出：页面、组件、路由、状态、接口（Mock 也可）
3. 指出需求里可能的冲突或遗漏

## 约束

- 只输出分析，不写代码
- 不明确处列出合理假设，供我确认
```

### 2. 拆任务（Step 2）

```markdown
## 背景

【同上，或引用 Step 1 分析结论】

## 已确认的需求清单

【粘贴 Step 1 输出】

## 请你帮我

按依赖顺序拆成 5–15 分钟可验收的小步，每步包含：

- 任务描述
- 涉及文件（预估）
- 验收标准（可观察结果）

## 约束

- 第一步固定为「路由 + 页面骨架」（如适用）
- 不要一步跨多层（路由 + 组件 + 状态混在一起）
- 只输出任务清单，不写代码
```

### 3. 代码生成（Step 4 · 主模板）

```markdown
## 背景

Vue 3 + Vite + Pinia + Vue Router，Composition API

## 当前相关文件

- 【填路径，如 src/router/index.js】

## 需求

【粘贴本步需求片段】

## 约束

- 使用 `<script setup>`
- 不引入新依赖
- 只改/新增：【列文件白名单】
- 禁止修改：【列黑名单，如 package.json、main.js】

## 本次只做

【明确本步范围，写清「不做」什么】

## 验收标准

- 【可观察结果 1】
- 【可观察结果 2】
```

**填写示例**（练习 1 · 路由 + 骨架）：

```markdown
## 背景

Vue 3 + Vite 项目，使用 Composition API、Pinia、Vue Router

## 当前相关文件

- src/router/index.js
- src/App.vue

## 需求

新增 /todos 路由和页面骨架，nav 增加 Todos 链接

## 约束

- 使用 `<script setup>`
- 组件放在 src/components/，页面放在 src/views/
- 不引入新依赖
- 只改/新增：src/router/index.js、src/views/TodosView.vue、src/App.vue

## 本次只做

- 注册 /todos 路由
- 创建 TodosView.vue 空壳（含标题「我的待办」）
- nav 增加链接
- 不做列表组件、不做数据

## 验收标准

- 访问 /todos 不白屏，能看到标题
- nav 可跳转 /todos
- pnpm dev 无报错
```

## 问题描述

【如：提取组件、简化重复逻辑、改命名】

## 约束

- 不改变现有功能和对外行为

### 6. 复盘（Step 6）

```markdown
## 背景

刚完成【练习 X / 任务 Y】

## 本次过程

- 任务拆分：（粘贴或简述）
- 用了几个 Prompt：
- 遇到的主要问题：

## 请你帮我复盘

1. 哪条 Prompt / 约束最有效？
2. AI 在哪里容易跑偏？怎么约束更好？
3. 时间分配是否合理？下次如何改进？
4. 有哪些约束应写入我的个人模板？

## 约束

- 只输出复盘建议，不改代码
```

### 7. UI/UX Pass（功能验收后 · 体验层）

```markdown
## 背景
【功能名】已跑通，现在只做 UI/UX 优化，不改业务逻辑。

## 设计规范（必须遵守）
- Tokens：`src/assets/base.css` 中 `--color-*` / `--btn-*` / `--input-*`
- 组件：`src/assets/components.css` 中 `.btn`、`.input`、`.status-tag`
- @src/assets/base.css @src/assets/components.css

## UX 要求
- 【如：主操作视觉权重 > 危险操作】
- 【如：新项插入列表顶部】
- 【如：输入框 focus 态清晰】

## 约束与范围
- 只改样式和布局相关代码
- 不改变 add/delete/校验等逻辑
- 禁止硬编码颜色、禁止 inline style
- 不引入 UI 库

## 验收标准
- 按钮 / 输入 / 标签全部走全局 class
- 无硬编码颜色
- 页面 5 秒内能看懂结构和操作
```

## 约束 FAQ

#### Q1 约束里的「新增」都要写详细地址吗？文件多了会不会太复杂？

按任务规模选写法：

1. **轻量**（1–2 个文件）
   - 只新增 `src/components/TodoList.vue`
   - 不要改其他文件
2. **中量**（3–5 个文件）
   - 允许新增/修改：`TodosView.vue`、`TodoList.vue`、`router/index.js`
   - 禁止修改：`package.json`、`main.js`、`App.vue`
3. **重量**（文件很多）
   - 只允许改动 `src/views/` 和 `src/components/` 下与 todo 相关的文件
   - 禁止修改：`package.json`、`main.js`、`vite.config.js`
   - 改动前先列出将要修改的文件清单

**原则**：白名单管「要做的事」，黑名单管「千万别碰的」。

#### Q2 「禁止改动」也要写很复杂吗？

不用。抓关键禁区即可：

- 不引入新依赖
- 不要改 `package.json`
- 不要改 `main.js`
- 不要改与本需求无关的文件

#### Q3 Prompt 拆到什么程度算合理？

粒度要和任务复杂度、时间、风险匹配 —— 不是 Prompt 越多越好。

- **练习阶段**：拆 4～5 步是为了练流程，不是为了最快交付
- **简单、熟悉的需求** → 1 个 Prompt 做完，完全合理
- **复杂、陌生、限时面试** → 才需要拆步
- 若永远按「任务 1 → 1.5 → 2 → 3 → 4」做，对简单题是过度工程，会违背「快」

#### Q4 「约束」和「本次只做」有什么区别？会不会重复？

**不重复，但简单题可以合并成一段。**

| | **约束** | **本次只做** |
|---|----------|--------------|
| 回答什么 | **怎么做、按什么规矩做** | **这一步做什么、不做什么** |
| 性质 | 规则 / 技术约定 | 任务范围 / 优先级 |
| 时效 | 往往多步通用 | 通常只针对当前这一步 |
| 例子 | script setup、不引新依赖、别动 package.json | 只做添加/删除，不做 Pinia |

- **约束** = 施工规范（用什么技术、哪些文件不能碰）
- **本次只做** = 今天的工单（这一步的功能边界）

同一条可以只写一处，例如「不要改 App.vue」：写在约束里表示长期规则；写在本步里表示强调这一步别顺手改。

**简单题（只改 1 个文件）可合并**，不必两块都写很长：

```markdown
## 约束与范围
- 只修改 src/components/TodoList.vue
- script setup，不引新依赖，不用 Pinia
- 只做添加/删除，不做编辑、不做持久化
```

**复杂题 / AI 爱多做时**再拆成「约束」+「本次只做」两段。

#### Q5 验收标准有必要写吗？不是人自己验收吗？

**对，最终一定是人验收。** Prompt 里的验收标准主要不是替代你点浏览器，而是 **告诉 AI「什么叫完成」**。

| 读者 | 作用 |
|------|------|
| **AI** | 知道做到哪停、少做多余功能（如顺手加编辑、弹窗） |
| **你** | 生成前想清楚；生成后对照检查 |

| | **Prompt 验收标准** | **人工必查清单**（见上文 § 验收必查清单） |
|---|---------------------|---------------------------------------------|
| 写给谁 | 主要是 AI | 主要是你 |
| 内容 | 本步功能 + 边界情况（如空标题不添加） | 能启动、能访问、无报错、样式可读 |
| 数量 | 2～4 条即可 | 固定 5 项 |

不写验收标准时，AI 容易多做功能，或漏掉「空标题不添加」这类细节。

**写法建议：**

- 有边界情况 → 写 2～3 条可观察结果（值得写）
- 功能极简、README 已写清 → 可合并进「需求」，或写「见 README 验收清单」
- 不必与人工清单重复堆砌；Prompt 管 **功能对不对**，人工清单管 **环境稳不稳**

**练习 2 示例（3 行够用）：**

```markdown
## 验收标准
- 输入标题添加后列表多一条，显示「未完成」
- 空标题点击添加，列表不变
- 点击删除后该条消失
```

#### Q6 改已有文件时，要不要写「保留」？

**会动旧逻辑时，应有一处写清「什么不动」** —— 不必须用「保留」作标题。

| 场景 | 建议 |
|------|------|
| 重构 / 迁状态（如 ref → Pinia） | 一句总括：原样迁入，除本 Prompt 增量外不改交互 |
| 以新建为主、旧文件小改 | 白名单 + 需求通常够用 |
| 本步要**改**的旧行为 | 写在「需求」，不要写进「保留」 |

等价写法：`本次只做`、`约束` 里「不改 xxx」、或「行为以 README §x 为准」。**不必**把上一步验收逐条复述进 Prompt。

```markdown
## 保留
将 TodoList 现有增删与列表逻辑原样迁入 Pinia；除本 Prompt 写明的新能力外，不改动已有交互与边界。
```

**本步新规则**（如「未完成」下标为完成 → 该项立刻从列表消失）写在**需求或验收**，不属于「保留」。

#### Q7 新 Prompt 还要重复上一步内容 / 样式规范吗？会不会越来越长？

**不必把已稳定的内容全文再抄一遍。** Agent 新对话没有「默认不动旧代码」的记忆，但靠**分层**即可，不必每步膨胀。

| 层次 | 放什么 |
|------|--------|
| 仓库 | `base.css`、`components.css`（长期规范） |
| README / STATUS | 练习需求、有效约束 |
| `prompt.md` | **本步增量** + 文件白名单 + **本步**验收 |

- **旧行为**：一句「保留 / 只迁不改」；细节以现有代码为准。
- **样式**：已有 design system 时，「参考删除按钮 / 沿用 components.css」常够用；无锚点的新 UI（如筛选 tab）可再加一句。
- **复杂业务**：规格书放稳定知识，Prompt 只写 **delta**。

跟练期 STATUS 写「Prompt 必引用 design system」——用**一行**满足即可，不必每练习粘贴整段 hex 禁令。

#### Q8 「保留」和「本次只做」有什么区别？

| | **保留** | **本次只做** |
|---|----------|--------------|
| 回答什么 | **已有实现别被带歪** | **这一步做/不做哪些能力** |
| 典型场景 | 迁 Pinia、提取组件、结构重构 | 不做持久化、不做 Pinia、不做编辑 |
| 和旧代码关系 | 默认**不变**（除非需求写明） | 定义**本步范围** |

简单题可合并成「约束与范围」一段；**迁状态 / 改旧文件**时建议单独有一句「保留」或等价表述。

> 六类边界详解 → 见 [`AI-coding-学习路线.md` § 三](./AI-coding-学习路线.md#三代码生成时的六类边界又快又好的关键)

## UI/UX 与 AI 协作

> **核心观点**：前端不可能只做功能、不管展示。Prompt 里「常不写细样式」≠ UI 可以凭感觉；而是 **第一刀优先功能，体验层靠规范 + 模板 + 验收**，不依赖个人经验。

### 两层工作流

```text
层 1 · 工程层   功能、数据、状态、边界  →  第一版 Prompt 主战场
层 2 · 体验层   布局、视觉、交互、一致性  →  有规范、有模板、有验收
```

### 四类设计资产（可沉淀、可复用）

| 资产 | 本项目位置 | 作用 |
|------|------------|------|
| **Design Tokens** | `src/assets/base.css` | 系统色、按钮/输入尺寸变量 |
| **组件契约** | `src/assets/components.css` | `.btn`、`.input`、`.status-tag` 等全局 class |
| **UI Brief** | 见下方摘要 / 自行维护 | 间距、密度、信息层级、禁止项 |
| **UX 规则** | 需求或 Prompt 中写明 | 新项置顶、空标题不添加、空状态等 **行为** |

**UI Brief 摘要（本项目）：**

- 主操作：`btn btn--primary`；危险操作：`btn btn--danger`
- 输入：`input`；状态：`status-tag--success` / `status-tag--warning`
- 禁止：硬编码颜色、每个组件重写 button 样式、inline style
- 列表：一条一行；表单区与按钮等高（`--btn-height` / `--input-height`）

### 实际开发三阶段

| 阶段 | 做什么 | Prompt 怎么写 |
|------|--------|---------------|
| **A · 功能** | 逻辑、结构、数据 | 主模板 + **一句 UI 约束**：「复用 components.css，禁止硬编码色值」 |
| **B · UI/UX Pass** | 视觉、交互、一致性 | 用 [模板 §7](#7-uiux-pass功能验收后--体验层) |
| **C · 验收** | 功能 5 项 + [体验 5 项](#体验-5-项uiux-pass-或上线前) | 人点浏览器确认 |

**阶段 A 功能 Prompt 里至少加一句（即使需求写「基本可读」）：**

```markdown
## UI 约束
- 复用 src/assets/components.css 中的 .btn / .input / .status-tag
- 使用 base.css 中的 CSS 变量，禁止 inline style 和硬编码色值
- 布局基本可读即可，视觉细节留 UI Pass
```

### 跟 AI 说 UI 的三种话术

| 级别 | 说法 | 适用 |
|------|------|------|
| **弱**（少用） | 「好看一点」 | AI 不可控 |
| **中** | 「用 .btn--primary，列表一条一行，新项置顶」 | 小改、对话修复 |
| **强**（推荐） | 「遵守 components.css 契约 + UI Brief，按体验 5 项验收」 | 正式开发、UI Pass |

### 什么时候第一版 Prompt 就要写 UI？

| 场景 | 做法 |
|------|------|
| 需求仅「基本可读」、练功能 | 第一版可只引用 design system；UI Pass 再细调 |
| 有设计稿 / Figma | 第一版附截图或标注 + 「严格按稿」 |
| demo / 客户导向 | 功能与展示同优先级，第一版写布局 + 组件 class |
| 项目已有 design system | 第一版就引用 `.btn`、tokens（本项目即此类） |
| 面试明确考 UI | Prompt 写清布局与全局 class |

### UX 行为 vs 视觉样式

| 类型 | 示例 | 写进第一版 Prompt？ |
|------|------|---------------------|
| **UX 行为** | 新 todo 置顶、空标题不添加 | ✅ 值得写（影响逻辑） |
| **视觉样式** | 绿色按钮、6px 圆角、focus 光晕 | 通常 UI Pass 或引用已有 class |

### 经验 vs 规范

| 经验（不可靠） | 规范（可靠） |
|----------------|--------------|
| 「按钮大概这种绿」 | `--color-primary` + `.btn--primary` |
| 「输入框现代一点」 | `.input` + focus ring token |
| 「验收时顺便看看」 | 功能 5 项 + 体验 5 项 |
| 「做完再调样式」 | 阶段 A → B 固定流程 + [模板 §7](#7-uiux-pass功能验收后--体验层) |

**经验决定「要什么感觉」；规范决定「怎么每次都说清楚、验清楚」。**

## 个人约束清单

> 每次 Phase 2 复盘后，把有效约束写在这里，下次 Prompt 直接复制。

<!-- 示例：
- 不要改 App.vue 的全局布局
- Todo 字段统一用 { id, title, done }
-->

#### Practice 1

- 简单静态页：1 个 Prompt 完成组件 + 引入
- 约束写：只新增 / 只修改 / 禁止改动（3 行）
- App.vue 只做 nav + RouterView，业务页不动全局布局
- nav 链接在 App.vue 统一覆盖全局 a 样式
- 样式小修：对话直接说，不必新建 prompt 文件
- Prompt 文件：每练习留最终版 + 复盘即可，草稿可删

#### Practice 2

- 功能 Prompt 聚焦逻辑；UI/UX 验收后迭代（按钮色、置顶、系统色、全局 .btn/.input）
- 样式/UX 沉淀到 base.css + components.css，后续 Prompt 引用而非重写
- UX 行为（空标题不添加、新项置顶）写进 Prompt；视觉细节走 UI Pass 或对话
- 简单题 1 个 Prompt；约束与范围可合并一段
- 「刷新恢复初始数据」= 无持久化，写 Prompt 时要表述清楚

#### Practice 3

- 迁 Pinia：用「保留」一句界定旧逻辑不动；新能力（toggle、筛选、筛选下 toggle 联动）写需求/验收
- 迁移来源写对文件（`TodoList.vue`，非壳页面）
- Prompt 写增量；样式「参考删除按钮」+ 一行 design system 即可

## Agent 协作（三角色 + 用户）

> 状态单一来源：[`STATUS.md`](../STATUS.md)  
> Cursor 规则：[`_templates/agent-workflow/.cursor/rules/`](../../../_templates/agent-workflow/.cursor/rules/)

| 角色 | 规则文件 | 做什么 | 能否写 STATUS |
|------|----------|--------|:-------------:|
| **需求拆解** | `requirements-analyst.mdc` | 分析 + **维护 STATUS** | ✅ |
| **Prompt 审** | `prompt-reviewer.mdc` | 只点评 Prompt | ❌ |
| **实现** | `implementer.mdc` | 按定稿 Prompt 写代码 | ❌ |
| **用户** | — | 定稿 Prompt、验收、复盘 | ✅ |

**标准流水线**：

```
原始需求 → [需求拆解]（更新 STATUS）→ 你写 prompt → [Prompt审] → 定稿 → [实现] → 验收 → 你更新代码现状
```

**模拟/限时**：Step 1 用需求拆解 Agent；中途不切换 Agent 角色。

**换 Agent 前**：更新或确认 STATUS → 新对话 `@tracks/ai-coding/STATUS.md` + 见 [`HANDOFF.template.md`](../../../_templates/agent-workflow/docs/agent/HANDOFF.template.md)

**prompt_status = 已定稿** 后，实现 Agent 才可改 `src/**`。
