# 用 AI 学 / 审 TypeScript 工作流

> AI 当教练，你当动手的人。最终类型文件以**你理解后的版本**为准。

## 原则

1. **先自己读 10 分钟**，再问 AI；带着具体疑问问，不要「整文件讲一遍」。
2. 解释阶段要求 AI **不改代码**；动手阶段你改，或让 AI 改但你必须能复述为什么。
3. 禁止把 `as any` / 乱 `as` 当默认修复；要求「最小修复 + 说明为何不是 any」。
4. 与 [`ai-coding`](../../ai-coding/notes/ai-coding-prompts.md) 衔接：审 diff 时多问「类型约束对不对」。

---

## 模板 1 — 解释（只解释，不改代码）

```text
背景：我在学前端 TypeScript，目标是读懂类型，不是让你重构。

请只解释，不要修改代码。

文件：<路径或粘贴片段>

请：
1. 用白话说明每个 type/interface 在约束什么（画出数据形状）
2. 标出危险的 `as` / `any` / 过宽的类型，并说明风险
3. 列出我可能卡住的 2～3 个点（用提问形式，先别直接给长答案）
```

---

## 模板 2 — 诊断（修报错）

```text
背景：Vue 3 + TypeScript。我要理解报错原因，不要用 any 糊弄。

报错原文：
<粘贴>

相关代码：
<粘贴最小片段>

请：
1. 用一句话说明根因（缺字段 / 联合未收窄 / 泛型不匹配 / …）
2. 给出**最小修复**（尽量少改行）
3. 说明为什么不应该用 `as any` 或盲目 `as`
4. 给我一道「改坏再修好」的变体题（先别给答案）
```

---

## 模板 3 — 出题（改坏再修好）

```text
根据我贴的文件，出 3 道 TypeScript 练习：
- 每道：你描述「如果改成 XXX 会怎样」
- 不要直接给修复代码
- 难度：1 入门 / 1 中等 / 1 稍难
- 文件：<路径或粘贴>

我做完后会把我的修法发你核对。
```

---

## 模板 4 — 用类型约束 AI 写代码（练习 11）

```text
背景：Vue 3 + <script setup lang="ts">，项目已有类型风格见 src/types/。

本次只做：
- 允许改的文件：<白名单，优先 types/*.ts>
- 明确不做：改业务逻辑、加依赖、用 any

类型要求：
- 定义：<接口名与字段>
- 调用方必须被类型约束（缺字段应编译失败）

验收：
1. pnpm exec vue-tsc --noEmit（或项目等价命令）通过
2. 无新增 any / 无无必要 as
3. 用 3 句话说明类型设计
```
